## 모바일에서 왜 그렇게 자꾸 깨졌나

제일 많은 시간을 쏟은 구간. 총 6가지 함정을 밟음.

### 1. 큰 글자를 숫자로 못 박아놓음

**문제**: CSS에 "타이틀은 280픽셀 크기"라고 숫자를 못 박아놓았음.
**비유**: 티셔츠 공장에서 XL 사이즈만 만들어놓고, S 사이즈 손님이 오면 옷에 몸을 우겨넣는 꼴. 옷은 펑퍼짐, 사람은 불편.
**결과**: 모니터에선 멋있게 큰 글자가, 아이폰에선 화면 밖으로 삐져나가서 잘려 보임.
**해결**: `clamp(최소, 화면비례, 최대)` 쓰기. "작은 화면에선 40px, 큰 화면에선 200px, 사이는 화면 비례로 자동 계산". 옷에 비유하면 신축성 있는 레깅스.

### 2. "반반씩 나누자"가 반반이 아니었던 문제

**문제**: CSS 그리드에서 `1fr 1fr`이라고 적으면 "반반씩 나눠라" 인 줄 알았는데, 아니었음.
**비유**: 형제 둘이 방을 반반 나누기로 했는데, 한 명이 피아노를 들여놓음. 피아노가 너무 커서 자기 방을 밀어냄. 다른 형제 방은 좁아짐.
**실체**: `1fr`은 "남는 공간을 1몫씩 나눠 가져라"인데, **속내는 "내 안의 내용물보다는 절대 작아지지 않음"**. 그래서 큰 글자가 든 칸이 부풀어오르면 다른 칸이 찌그러짐.
**해결**: `minmax(0, 1fr)`로 바꾸기. "내용물이 아무리 커도 1몫만큼만 가져가라"는 엄격한 반반 규칙. CSS 그리드 쓸 때 필수 공식.

### 3. 이름표를 엉뚱한 사람 명찰 안쪽에 붙임

**문제**: CSS 그리드는 "A 칸에 뭘 놓을지" 지정해야 하는데, 그 지정을 **엉뚱한 깊이**에 걸어놨음.
**비유**: 회사에서 "박 부장은 4번 자리"라고 정해놓고, 막상 박 부장의 **책상 서랍 안**에 '4번 자리' 이름표를 붙여둔 꼴. 회사에선 박 부장이 어디 앉을지 모르니까 아무데나 앉혀버림.
**실체**: HTML 구조가 `<바깥박스><안박스>내용</안박스></바깥박스>`인데, `grid-area`를 안박스에만 붙였음. CSS 그리드는 **직접 자식**(바깥박스)만 보는데.
**해결**: 바깥박스에 `grid-area`를 붙이기. 혹은 `:nth-child(1)` 같은 순서 지정도 OK.

### 4. 페인트통 라벨만 바꿨더니 벽이 안 보임

**문제**: "우리 회사는 이제 파란색만 쓴다"라는 의도로 원래 있던 '코랄색' 페인트통에 파란색을 부었음.
**비유**: 그런데 옛날 설계도엔 "벽은 코랄 페인트, 그 위에 코랄 테두리 원을 그려라"라고 되어 있었음. 설계도 그대로 쓰니까 **파란 벽에 파란 원** = 원이 안 보임. Day 3 타이틀 글자 위에 같은 색 원이 겹쳐서 글자가 사라졌던 이유가 이거.
**교훈**: 색 이름을 하나로 통일할 땐, **그 색이 어디서 어떻게 쓰이는지 전부** 확인해야 함. 그냥 페인트통만 바꾸면 과거의 디자인 의도가 깨짐.
**다음엔**: 색 이름을 "파랑", "코랄"처럼 색으로 짓지 말고 **용도**로 지을 것. 예: "배경색", "강조색", "테두리색". 그러면 어디 쓰는지 이름만 봐도 앎.

### 5. 아이폰이 옛날 것을 기억함

**문제**: 코드를 배포하고 아이폰에서 새로고침해도 이전 버전이 보임.
**비유**: 냉장고가 일주일 전 음식을 너무 오래 기억해서, 오늘 새로 넣은 음식을 달라고 해도 자꾸 옛날 것만 꺼내줌.
**실체**: 아이폰 Safari는 웹페이지의 CSS·JS를 오래 캐시에 붙들어둠. 게다가 아이폰의 모든 브라우저(크롬, 파이어폭스 포함)는 다 Safari 엔진을 씀. 즉 크롬 깔아도 똑같음. (애플 정책)
**해결**: HTML에 "이 페이지는 캐시하지 말아달라"는 메타 태그 추가. 그리고 사용자한테는 "주소창 내려서 pull-to-refresh 하면 강제 새로고침됨"이라고 알려주기.

### 6. 항상 떠 있는 단추가 내용을 가림

**문제**: 화면 하단에 고정된 툴바(스크롤 버튼, 다운로드 등)가 페이지 맨 아래 "다음 페이지" 버튼을 덮어버림. 눌러지지 않음.
**비유**: 식당 테이블 위에 메뉴판을 풀로 고정해놨는데, 음식 접시 놓을 자리랑 겹침. 음식 나오면 메뉴판에 가려져서 뭘 먹는지 안 보이고, 먹기도 불편.
**해결**: "다음 페이지 버튼이 화면에 보일 때만 툴바를 자동으로 아래로 내리기". 음식 나오면 메뉴판이 알아서 비켜주는 식.
**기술명**: `IntersectionObserver` (요소가 화면에 보이는지 감시하는 브라우저 내장 기능).

## 콘텐츠 검증 — 링크가 엉뚱한 곳으로 가던 이유

### "Sunset Beach"가 호텔로 연결됐던 사건

**문제**: 가족들이 SUNSET 카드 누르면 구글맵이 "SUNSET 호텔"을 보여줌. 우리는 해변을 기대했는데.
**비유**: 외국 손님에게 "김씨 아저씨 가게 어디냐"고 물으면 한국에 김씨 아저씨가 너무 많아서 엉뚱한 가게로 안내되는 것.
**실체**: "Sunset Beach"는 전 세계 수백 곳에 있음. 그냥 그 단어만 던지면 구글이 아무거나 고름.
**해결**: **공식 명칭으로 검색어 고정**. "Chatan Park Sunset Beach" 처럼 정부에서 지정한 정확한 이름으로. 코드에 장소별 검색어 사전(`PLACE_QUERIES`)을 만들어서 클릭 시 정확한 검색어가 전송되도록.
**교훈**: 검색 엔진에 "알아서 찾아줘" 하지 말고, **정확한 이름을 직접 지정**. 게으르게 쓰면 게으른 결과 나옴.

### "근처 소바집" 같은 애매한 표현 금지

**문제**: 처음엔 "추라우미 근처 소바집" 이라고 막연하게 적어놨음. 링크 걸 수도 없고, 가족이 봐도 어딘지 모름.
**교훈**: 장소는 **고유명사**로. "1905년 창업 Kishimoto Shokudo" 정도는 돼야 검증도 되고 구글맵도 찾아냄.

### 주소는 건물명·번지까지

**처음**: "Depot Island Seaside 3F, 호텔 도보 3분"
**정확**: "depoisland seaside 3F · 미하마 9-21 · 호텔 바로 옆"
호텔이 미하마 9-22, Calif Kitchen이 미하마 9-21. **옆 건물**. 처음 썼던 "도보 3분"은 틀림.

## 보안 — "가족용인데 이런 걸 왜 해?"

### 하나밖에 없는 공격 경로

이 앱에는 "↓ 가져오기" 버튼이 있다. JSON 파일 하나 받으면 여행 기록 전체를 그걸로 덮어씀. 편리하지만, 바로 여기가 유일한 공격 창구였다.

**문제**: 원래 코드는 받은 JSON 안의 사진 주소를 HTML 문자열에 그대로 갖다 붙였음.
**비유**: 택배 상자 안에 뭐가 들었는지 안 보고 그냥 열어서 식탁에 통째로 쏟아버리는 것. 정상 택배면 괜찮은데, 가끔 폭탄이 들어있다.
**실체**: 교묘하게 만든 "사진 주소" 문자열이 HTML 속성의 따옴표 틈을 뚫고 **임의의 자바스크립트로 둔갑** 가능. "가족이 보낸 백업이야~" 하면서 악성 JSON을 피해자가 가져오면 그 사람 브라우저에서 실행됨.
**실제 영향도**: 이 앱엔 로그인·개인정보·돈 없음. 빼내봐야 남의 여행 메모 정도. 실제 위협은 낮았음.

### 4겹 방어로 고침

비유: **택배 검수 규칙**.

1. **받은 상자 그대로 쓰지 말고, 내용물 하나씩 꺼내서 새 상자에 담기**
   `sanitizeImportedState()` 함수가 허용된 필드만 옮기고 나머지는 폐기. 이름이 이상한 필드, 값이 이상한 필드 전부 버림.

2. **"사진"이라는 상자는 진짜 사진인지 포맷 검사**
   사진 데이터는 반드시 `data:image/...` 형식으로 시작해야 하고, 용량 상한도 있음. 아니면 거절.

3. **HTML은 문자열 합치기 금지, 브라우저 API로 만들기**
   원래 코드: `"<img src='" + 받은데이터 + "'>"` ← 이 방식이 위험
   고친 코드: `img = createElement; img.src = 받은데이터` ← 브라우저가 자동으로 따옴표 이스케이프
   비유: **손님이 써준 메모를 메뉴판에 풀로 바로 붙이지 말고, 메뉴판 시스템에 입력.** 시스템이 알아서 이상한 문자 처리함.

4. **특수 키는 바로 거절**
   JSON 안에 `__proto__` 같은 자바스크립트 특수 키가 있으면 즉시 폐기. 놔두면 앱 전체 동작이 오염될 수 있는 열쇠.

### 왜 고쳤나 — "가족만 쓰는데?"

진짜 이유는 **도쿄 여행 개인용 버전 복제 예정**이라서. 지금 원본에 있는 구멍은 복사본에도 그대로 따라감. 개인용은:
- 본인만 쓰니까 공격 기회는 여전히 적음
- 대신 들어갈 데이터가 더 민감해질 수도 있음 (예약번호, 개인 일기, 지출)
- 혹시 나중에 링크를 블로그·SNS에 흘리면 순식간에 공개 상태

**교훈**: "지금은 안 위험해"는 맞지만, **코드는 언젠가 복제·공유됨**. 원본에 구멍 있으면 사본도 그대로 뚫린 채로 복제됨. 지금 10분 투자 = 미래의 모든 사본이 안전.
**다음엔**: 새 여행지 버전 만들 때 sanitizer 먼저 옮기고 콘텐츠 작업. 순서 바꾸지 말 것. 새 state 필드 추가할 때마다 sanitizer에 대응 블록 같이 추가.

## 숙소를 잘못 알고 있었던 사건 — "현지에선 못 고친다"

### 발단

매거진 거의 완성된 시점, 가족과 일정 확인하다가 알아차림: **4박 내내 차탄 Vessel 호텔에 있는 줄 알았는데 실제로는 북부 Orion 2박 + 차탄 Vessel 1박**이었음. 매거진은 전부 "호텔이 아메리칸 빌리지 안이라 도보 1분..." 전제로 쓰여있었음.

**비유**: 결혼식 청첩장을 다 돌렸는데 예식장 주소가 틀린 거랑 비슷함. 한 번 인쇄·배포되면 수정 못 하니 그 전에 잡아야 함.

### 왜 이게 큰 문제였나

"현지에선 수정 불가"라는 제약이 모든 걸 바꿈:
- 여행 중 "이거 주소 틀렸네" 발견 → 당황 → 검색 → 시간 낭비
- 구글맵 링크가 엉뚱한 곳으로 → 엄마가 혼자 길 잃음
- 식당 영업시간 착오 → 문 닫힌 가게 앞에서 가족 전체 당황

## 두 세션이 같은 벽을 칠하면 — 도쿄 폴더 충돌 사건

### 무슨 일이 있었나

도쿄 매거진을 만들겠다고 오키나와 채팅방에서 도쿄 폴더에 셸을 만들기 시작했다(오키나와 6500줄 코드 복사 + 식별자만 도쿄로 교체). 17:25–17:28에 작업 완료.

근데 같은 시간, **다른 채팅방**에서 도쿄 PRD v0.2 기반으로 v1을 빌드하고 있었다 (single-page Quick Capture · IndexedDB · 1700줄 슬림 버전). 17:41에 그쪽 세션이 `index.html`을 v1으로 덮어씀.

결과:
- 내가 만든 셸 → 사라짐 (덮어쓰기 당함)
- 시간 낭비 + 사용자 혼선
- 다행히 git이 따로 있는 폴더가 아니라 충돌 자체는 안 났지만, 큰 작업이었으면 한 쪽 일이 통째로 날아갈 수 있었음

### 비유

**이사 중인 집에 짐꾼 두 팀이 따로 와서 같은 가구를 옮긴 셈**. 둘 다 일을 하고 있었지만, 한 팀은 다른 팀이 이미 옮기고 있다는 걸 모름. 결국 한 팀의 작업은 의미 없어짐.

또는 **같은 벽에 두 사람이 다른 색 페인트를 동시에 칠하는 것**. 마지막에 칠한 사람이 이김. 먼저 칠한 사람의 작업은 그대로 사라짐.

### 왜 이게 일어났나

- **폴더 단위 잠금 메커니즘이 없음** — git이 있는 곳이면 동시 편집이 충돌로 잡히지만, git 없는 폴더는 그냥 마지막 저장이 이김
- **Claude Code 세션 간 통신이 없음** — 다른 세션이 뭘 하고 있는지 서로 모름
- **사용자가 두 채팅방에서 같은 폴더를 열어줬음** — 이게 발생 조건

### 교훈

1. **한 폴더 = 한 세션**. 부모 README에 명시 룰 추가됨 ("⚠️ 여러 Claude Code 세션이 같은 매거진 폴더를 동시에 작업하지 말 것").
2. **다른 세션이 진행 중인 폴더에는 직접 파일 안 건드리고 명세서로 넘기기**. 코드 직접 수정 = 충돌. 대신 다음 패턴:
   - 부모 폴더에 `XXX_TRANSPLANT_TO_YYY.md` 같은 이름의 명세서 파일 작성
   - 명세에는 "어디 코드를 어떻게 변형해서 가져갈지" 자세히
   - 다른 세션이 이 파일 읽고 작업
   - 결과는 다른 세션이 자신의 `learning.md` / `CLAUDE.md`에 기록
3. **명세서를 부모 폴더에 두는 이유**: 두 폴더(소스·대상) 모두에서 참조 가능. 어느 한 폴더에만 두면 다른 쪽 세션이 못 보거나 cwd 충돌.
4. **시작 전 modification time 확인**. 폴더에 누군가 최근 작업한 흔적이 있으면(파일 timestamp가 분 단위로 새것이면) 일단 멈추고 상황 파악.

### 다음에 비슷한 일 할 때

**A 세션이 B 폴더 작업에 기여하고 싶으면**:
1. B 폴더 파일 직접 수정 ❌
2. 부모 폴더에 명세서 파일 작성 ✅
3. 사용자에게 "B 채팅방에 이 파일 가리키고 작업 시켜" 안내
4. B 세션이 적용 후 명세서 진행 상황 체크박스 갱신
5. 나중에 명세서 자체는 archive 폴더로 이동하거나 그대로 두고 참고용

**예시 (실제)**: `UI_TRANSPLANT_TO_TOKYO.md` (2026-04-25 작성) — 오키나와의 인터랙션·툴 13개를 도쿄 v1에 가져갈 때 어떻게 변형할지 Tier 1/2/3로 분류한 명세서.

### 한 줄 요약

> **여러 채팅방에서 같은 폴더를 열지 말 것. 어쩔 수 없이 다른 세션 영역에 기여해야 하면, 직접 손대지 말고 명세서로 넘겨라.**

## 왜 React 안 썼냐

간단하게 비교해봄:

| 상황 | vanilla (지금 방식) | React |
|---|---|---|
| 한 파일로 끝내고 싶음 | 완벽 | 여러 파일 + 빌드 단계 필요 |
| 가족이 수정하다 깨뜨릴 여지 | 거의 없음 | package.json 잘못 건드리면 사이트 다운 |
| 호텔 Wi-Fi 불안정해도 작동 | 한 번 받으면 오프라인 OK | 번들 받아야 함 |
| 새 여행지 버전 만들기 | 파일 복사 + 내용 교체 | 컴포넌트 재사용 가능 |

**비유**: 동네 분식집 차리는데 공장을 지을 필요 없음. 분식집은 분식집답게 간단한 게 답. 나중에 "전국 체인 100개 내겠다" 하면 그때 공장 고민.

지금 이 프로젝트는 **딱 분식집 규모**. React는 과함.

## § 3. 미션 시스템 — "9개" 가 9개가 아닐 뻔한 사연

> 가족용 매거진에 게임화 요소를 넣다가 4번 회귀를 겪은 이야기.

### 미션이 두 종류여서 헷갈렸다

매거진 안에 도장 찍는 곳이 두 군데 있음:
- **독립형 미션 카드** (`.mission-card`) — 호텔 도착, 가족 사진, 비행기 탑승 같이 "별도 카드"
- **옵션 안 미션** (`.opt[data-mission]`) — 추라우미 수족관 같이 "고를 수 있는 옵션이면서 미션"

총 9개 미션인데 처음엔 둘을 같은 방식으로 다뤄서 이상한 일이 생겼음.

**문제**: "추라우미 수족관" 카드를 ✓ 누르면 — 도장도 찍히고 동선에도 들어감. 한 클릭에 두 일이 동시 처리. 그런데 호텔 도착 미션은 도장만 찍히고 동선에는 안 들어감 (호텔은 항상 동선에 있어서). 같은 "미션" 인데 동작이 달라 보임.

**비유**: 학교 출석부에 도장 찍는 거랑, 운동장 출석 체크판에 도장 찍는 거. 둘 다 "출석"이지만 운영 방식 다름. 학생한테 "둘 다 출석이야"라고 말하면 헷갈림.

**해결**: 두 종류는 본질적으로 다름을 인정. 별도 코드 경로:
- 옵션형 (.opt) → 사용자 선택이 곧 미션 = `state.chosen` 과 `state.missions` 동시 토글
- 독립형 (.mission-card) → 도장만 찍는 행동 = `state.missions` 만 토글
- 독립형이 동선에 영향 줄 때는 `midHotel.afterMission` 으로 별도 정의

### 9개 인덱스가 단일 source of truth 안 되면 망함

미션 번호 1~9 가 **세 군데에서 표시**됨:
- 미션 카드 좌측 상단 라벨 ("Mission 3 · Check-in 15:00")
- 옵션형 카드 별도 pill ("Mission 5 · 고래상어")
- 마무리 페이지 도장 그리드 좌상단 동그라미 (1, 2, 3, ...)

**문제**: 처음엔 각자 따로 번호 매김. 미션 추가하니까 어떤 곳은 1~9, 어떤 곳은 1~10, 어떤 곳은 mission ID 알파벳 순. 가족이 "Mission 5 어디 있냐" 물으면 헷갈림.

**해결**: `getAllMissions()` 라는 함수 하나가 **DOM 순서**로 9개 정렬해서 반환. 모든 곳이 이 함수만 참조 (single source of truth). DOM 에서 미션 위치 바꾸면 자동으로 번호 다시 매겨짐.

**교훈**: "같은 의미의 번호" 가 두 군데 이상에 나오면 — **둘 다 같은 함수에서 받아오게 하라**. 안 그러면 한 쪽만 갱신해서 어긋남.

### 마지막 미션 클리어 시 폭죽 vs 도장만

**문제**: 9번째 미션 도장 찍으면 — 일반 도장 애니메이션 (1.8초) 만 발동. 9개 모은 보상 X.

**해결**: `celebrateMission()` 안에서 `allDone` 검사. 마지막이면 `fireTripComplete()` 호출 (3.2초 + 60개 confetti + 도-미-솔-도 사운드). 마무리 페이지로 자동 이동 + 인증 카드 화려한 등장.

**교훈**: 게임화 핵심 = "마지막 보상이 1번째와 달라야 한다." 똑같은 도장이면 9번째 모은 의미가 약함.

### Out-of-order 클릭 안전성 (회귀 조심)

**문제**: 가족이 9개 순서대로 안 누름 — 7번 먼저, 3번 그 다음, 1번 마지막. 마지막에 누른 게 1번이면 trip-complete 발동? Yes — 그게 정답. 어떤 순서든 9번째 done 카운트 도달 시점에 발동.

**검증**: 9개 무작위 순서로 클릭하는 자동 테스트 (browser console). 마지막 클릭마다 trip-complete 발동 확인.

## § 4. 동선 시스템 — 호텔 체크인을 동선에 끼우기

> 단순해 보이는데 회귀가 가장 자주 나는 영역.

### "Mid Hotel" 메커니즘 — DOM 위치가 곧 시간 순서

D1 (공항 → 북부 호텔) 같은 이동일에는 호텔 체크인이 하루 중간 이벤트. 동선 chip 으로 표현하면:
```
공항 (Start) → Blue Seal 12:30 (점심) → Orion 호텔 체크인 15:00 → 일몰 18:00 → Orion 호텔 (End)
```

**문제 1 (초기)**: 호텔 체크인 chip 을 항상 chip #2 위치에 강제 삽입. 그러면 Blue Seal (chip #2 가 되어야 할) 이 chip #3 으로 밀림. 시간 순서랑 안 맞아.

**해결 1**: `computeMidInsertIdx()` 함수 — 미션 카드의 **DOM 위치** 를 기준으로 "이 픽보다 앞이면 mid 앞에, 뒤면 mid 뒤에" 자동 계산. DOM 순서 = 시간 순서로 작성하면 자동으로 올바른 위치.

**문제 2 (회귀)**: D1 미션 #2 호텔 도착 카드를 페이지 위쪽에 두고, Blue Seal 옵션을 그 아래에 둠. DOM 상 Blue Seal 이 미션보다 뒤. 그러면 computeMidInsertIdx = 0 (mid 앞에 픽 0개) → 호텔 chip 이 chip #2 위치 (Blue Seal 보다 앞).

→ 시간 순서랑 또 안 맞음.

**해결 2**: 미션 카드 위치를 Blue Seal 옵션 **뒤**로 이동. 그러면 자동으로 맞춰짐.

**교훈**: DOM 작성 순서가 곧 시간 순서. "기획상 어떤 순서로 일어나나" 를 그대로 HTML 에 박아야 동선 시스템이 자동 처리.

### "3-call-sites consistency rule" — 한 곳 고치면 다른 두 곳도 봐야

`midInsertIdx` 를 쓰는 함수가 **3 곳**:
1. `renderDayRoute(dayId)` — chip 가로 흐름 박스
2. `renderDayMap(dayId)` — Leaflet 폴리라인 + 마커
3. `assignPickNumbers(dayId)` — 옵션 카드 좌상단 번호

세 함수 결과가 어긋나면 사용자가 chip 5 = 카드 5 = 지도 마커 5 매핑 못함. 회귀 사례 (2026-04-27): D3 mid 가 fixed position 2 였을 때 chip / opt 번호가 어긋났음 → DOM-based midInsertIdx 로 통일.

**룰**: 동선 mid 로직 변경 시 → 3 군데 동시 수정. 안 그러면 silent regression. ROUTE.md § 3-call-sites consistency rule 명시.

### "Ogasawara vs Okinawa" — 좌표 한 번에 안 본 사연

마무리 페이지 일본 지도에 오키나와 강조 ★ 박을 때 처음엔 좌표 (381, 535) 를 사용. 형이 "오키나와가 동남아에 있냐"는 피드백.

**문제 분석**: 일본 지도 SVG 의 Präfekturen path 좌표를 클러스터 분석하니까 — 남쪽에 두 클러스터 있음:
- 클러스터 A: x=19~183, y=499~549 → **진짜 오키나와 (류큐 제도)**
- 클러스터 B: x=379~397, y=492~549 → **오가사와라/이즈 제도** (도쿄도 산하, 태평양 1000km 남쪽)

도쿄 (35.7°N, 139.7°E) 가 SVG 상 x≈380. 같은 경도에서 남쪽 1000km = 오가사와라. 진짜 오키나와는 **남서쪽** (x 작아지고 y 커지는 방향).

**해결**: 좌표 (381, 535) → (105, 502) 로 정정. SVG file viewBox 도 타이트 크롭 (10 40 470 530) 해서 한반도·중국 제거.

**교훈**: 지도 SVG 에 좌표 박을 때 — 추측 X. **Präfekturen 그룹의 path 좌표 클러스터 분석** 으로 진짜 위치 확정. 하나의 큰 데이터 더미에서 좌표를 뽑을 땐 패턴 (도쿄 ~ 오키나와 위치 관계) 부터 검증.

### "Day 1 호텔 인포 카드" 가 미션 밑에 묻힘

운전자(자형/형) 가 D1 페이지 열면 호텔 주소·전화번호·매피온 코드가 필요. 그런데 처음엔 호텔 카드가 미션 #2 (호텔 도착) **밑** 에 있어서 가족 첫 사진 미션까지 스크롤해야 보임.

**해결**: 호텔 인포 카드를 미션 #2 **위** 로 이동. 박스 간격은 .day flex gap 그대로 유지 — hotel-card 자체 margin 변경 없이 단순 DOM swap.

**교훈**: 페르소나 (운전자) 사용 시나리오 (운전 중 빠른 주소 검색) 를 떠올리면 — 정보 우선순위가 보임. "어른은 미션 안 누름" 이라는 페르소나 가정이 결정의 근거.

### 처음엔 CLAUDE.md 한 곳에 모든 룰을 우겨넣었다

`CLAUDE.md` 한 파일에:
- 여행 사실 (일정, 호텔, 비밀번호)
- 스택·파일 구조
- 7페이지 구조
- 인터랙션 룰 (페이지 전환, localStorage, 장소 클릭)

- 장소 검증 프로세스 + 출처 강도
- state 3-namespace 룰
- 번호 시스템 (지도 vs 추가옵션)
- 디자인 토큰
- 자주 하는 작업 (장소 추가, 페이지 추가, 새 버전)

이 문서가 200줄을 넘어가면서 — Claude 가 매번 찾아야 함. 형도 "어디에 뭐 있는지" 파악 어려움. 작업할 때 어떤 룰이 적용되는지 추적 불가.

**문제 신호**: 형이 "ROUTE 동선 md 가 계속 어긋나기만 하는데?" 라고 말함. 동선 룰을 변경했는데 다른 영역에서 같이 봐야 할 룰을 못 보고 회귀.

### 위계 1차 시도 — 도메인별 분할 (DESIGN/ROUTE/MISSIONS)

CLAUDE.md 의 도메인별 룰을 분리:
- `DESIGN.md` — 아키텍처·상태·디자인 토큰
- `ROUTE.md` — 동선 시스템
- `MISSIONS.md` — 미션 시스템
- `VERIFICATION.md` — 회귀 검증
- `DOC-MAP.md` — 마스터 인덱스

**그런데도 문제 발생**: 형이 "CLAUDE.md 가 왜 저기 가있어? 단일 source of truth 는 그렇다 쳐도 CLAUDE.md 가 그래도 바로 그 밑에 와야 하는거 아니냐?" 라고 지적.

→ 모든 docs 가 평면적으로 펼쳐져 있고 위계 (Level) 가 없음. CLAUDE.md 가 "운영 룰북" 인데 다른 도메인 docs 와 같은 무게로 보임.

→ TODO.md 가 CLAUDE.md 만 참조. learning.md 가 Level 1 옆에 평행으로 보여서 회고가 작업 가이드와 같은 무게.

→ 비전 문서 (CONCEPT/PRD) 자체가 없음. "왜 만드나" 가 어디 적혀있지도 않음.

### 위계 2차 — 5단 Layer 체계

```
L0 VISION         CONCEPT.md (기획서) + PRD.md (요구사항)
                    ↓
L1 PROJECT GUIDE  CLAUDE.md (Claude 작업 진입점)
                    ↓
L2 IMPLEMENTATION DESIGN.md, ROUTE.md, MISSIONS.md (도메인별)
                    ↓
L3 OPERATIONS     VERIFICATION.md, TODO.md
                    ↓
L4 KNOWLEDGE      learning.md (이 문서, Level 0~3 종합)

INDEX:            DOC-MAP.md (전 layer 인덱스 + 영향 매트릭스)
```

**효과**:
- 위에서 아래로: 비전 → 운영 룰 → 구현 명세 → 검증 → 회고
- 코드 만지기 전에 → DOC-MAP.md 영향 매트릭스 → 어떤 docs 같이 봐야 하는지 파악
- 새 미션·픽·페이지 추가 → 영향 받는 docs 동시 갱신
- 모든 docs 끝에 "참조" 섹션으로 위·아래 link

### 고아 (orphan) 문서 점검

위계 정리 후 cross-reference 매트릭스:
- 각 doc 이 다른 doc 들에 의해 몇 번 참조되는지 카운트
- 0번 참조 = 고아 (orphan) → 위계 깨짐 신호

**찾은 문제** (2026-04-28):
- `DESIGN.md`, `VERIFICATION.md` 끝에 "참조" 섹션 없음 → 다른 docs 에서 참조하는데 본인은 어디로 link 거는지 명시 X
- `DOC-MAP.md` 가 CLAUDE/TODO/learning 3 곳에서만 참조 → 마스터 인덱스인데 도메인 docs 가 link 안 거는 게 이상함

**해결**: 모든 docs 끝에 "참조" 섹션 표준화 — 위 (Level 0,1) / 옆 (같은 Level) / 아래 (Level 3,4) / 인덱스 (DOC-MAP.md) link.

### 교훈 — "위계가 없으면 작업 자체가 시간 손실"

비개발자 메인테이너 (형) 가 한 말: "기본적인게 왜 안 되지?" — 그 원인 추적해보면 **항상 문서 위계 깨진 곳**. 한 곳을 만지고 다른 곳을 안 만짐.

**룰**:
1. 새 문서 만들 때 → Level 결정 → DOC-MAP.md 등록
2. 코드 만질 때 → DOC-MAP.md 영향 매트릭스 확인
3. 큰 결정 변경 → Level 0 (CONCEPT/PRD) 까지 거슬러
4. 분기 1회 → cross-reference matrix 점검 (orphan 찾기)

**한 줄 요약**: 위계 없는 문서는 zip 파일 안에 흩어진 README들이랑 같음. 위계 = 문서 간 관계.

## § 7. UI ↔ Canvas Parity — "다운로드 이미지가 화면이랑 다른데?"

> 같은 컴포넌트를 두 군데에서 따로 그리는 게 가장 흔한 회귀 원인.

### 마무리 페이지 인증 카드는 두 번 그려진다

`trip-conquer-card` (오키나와 정복! 인증 카드) 는:
- **화면 UI** — HTML/SVG/CSS 로 동적 렌더 (`.td-conquer`, `.td-stats` 등)
- **다운로드 이미지** — Canvas 로 따로 그림 (`downloadConquerImage()`)

가족 단톡 공유본 (PNG) 이 화면과 다르면 → "내가 본 거랑 다른데?" 혼란.

### 회귀 사례 1: 오키나와 줌 SVG 누락 (2026-04-28)

화면 UI 에 일본 지도 밑에 **오키나와 본도 줌 박스** + **줌인 화살표** 추가했음. 그런데 Canvas `downloadConquerImage()` 는 그 부분 안 그림 → 다운로드 이미지에 오키나와 부분이 없음.

**형의 피드백**: "야 그리고 여기서 이미지 저장하면 오키나와 부분이 표기가 안되는 거 같은데 문제 해결해줘. 그리고 이런 부분은 design.md 에 뭔가 문제가 있어서 안되는거야? 왜 기본적인게 구현이 안되지?"

→ 정확함. design.md 에 "dual-render parity" 룰이 없어서 한 쪽만 만지고 다른 쪽 까먹음.

**해결**:
1. `drawOkinawaZoomToCanvas()` 함수 신설 — okinawa-mainland.svg 비동기 로드 + cream 톤 보정 + dashed border
2. `downloadConquerImage()` 안에 호출 + 화살표 그리기 추가
3. Canvas 높이 1240 → 1560 (오키나와 줌 자리)

### 회귀 사례 2: 통계 값이 다 비어있는 다운로드

**형의 피드백**: "아니 저장을 했는데 왜 수치들이 안 나오고 다 날라가있냐?"

→ 9/9 가 아니라 0/0 / — / 0 / 0 으로 다운로드됨.

**원인**: `updateTripStats()` 함수가 stat DOM 값을 채우는데, 이 함수는 **9개 미션 모두 done 시점에만** 호출됨 (`updateMissionCounter()` 안의 `if (allDone) updateTripStats();`). 그 전 시점에 download 버튼 누르면 — DOM 의 [data-stat] 값들은 HTML 초기값 (0/0, —, 0, 0) 그대로.

Canvas 는 DOM 에서 `textContent` 읽어서 그냥 그림. 그래서 stale 값이 PNG 에 박혀나옴.

**해결**: `downloadConquerImage()` 시작 부분에 `updateTripStats()` 강제 호출 — 항상 fresh 값 보장.

### 룰 — "Dual-render Component" 들 식별 + parity 강제

`MISSIONS.md § UI↔Canvas Parity Rule` + `DESIGN.md § Dual-render Components` 로 명문화:

| 컴포넌트 | 화면 (HTML/SVG) | Canvas | 동기 룰 |
|---|---|---|---|
| 인증 카드 | `.trip-conquer-card` | `downloadConquerImage()` | UI 변경 시 canvas 도 같은 commit 에 |
| 방문한 곳 리스트 | `<details class="visited-list">` | `downloadVisitedListImage()` | 같음 |

**룰**:
1. 화면 시각 요소 추가/변경 → canvas drawing 에도 동시 적용
2. 두 코드 위치를 **같은 commit** 에 묶어 PR (분리 commit 금지)
3. **항상 fresh state**: canvas 가 DOM 읽기 전에 update 함수 호출 (updateTripStats 등)
4. VERIFICATION.md [P] 항목으로 시각 비교 검증

**교훈**: "한 컴포넌트를 두 군데서 따로 그리면 — 한 쪽만 갱신해서 어긋난다" 는 자명한 함정인데도 우리는 두 번 빠졌음. **dual-render 컴포넌트 식별** + **parity rule 문서화** 가 유일한 방어선.

## § 8. 비개발자 메인테이너와 일하기 — "코드 못 짜는 형의 입장"

> 형은 개발자가 아니다. 매번 "이거 왜 안 돼?" 를 물어볼 사람 없다. learning.md 가 마지막 안전망.

### 메인테이너의 처지를 가정하고 만들어야

이 프로젝트의 메인테이너 (형) 는 코드를 직접 짤 수 없음. Claude 같은 AI 어시스턴트에게 자연어로 요청하고 결과를 검증함.

이 시나리오가 의미하는 것:
1. **모든 의사결정은 자연어 문서에 남아야** — "왜 이 결정?" 추적 가능해야
2. **회귀 패턴을 한국어로 적어 둬야** — 다음 Claude 세션이 같은 함정 안 빠지게
3. **시각 검증 가능해야** — 코드 안 봐도 "동작이 맞나" 폰에서 확인 가능
4. **메인테이너의 피드백 = 시그널** — "왜 안 돼" 는 곧 "위계/문서 어디 깨졌나" 의 신호

### 형이 한 자주 묻는 질문 (FAQ 식)

> **"왜 이거 기본적인게 안 되지?"**
→ 거의 항상 dual-render parity (§ 7) 또는 위계 깨짐 (§ 6) 이 원인. 자기 처음 갈 때 양쪽 다 점검.

> **"갑자기 이상하게 보이는데?"**
→ 모바일 / 캐시 이슈 (§ 1) 또는 state 3-namespace 누락 (§ 2). 폰 hard-refresh + 자동 검증 첫 점검.

> **"위계가 안 잡혔는데?"**
→ DOC-MAP.md 마스터 인덱스 보고, 새 변경이 어느 Level 에 속하는지 분류. orphan 점검.

> **"코딩 과정에서의 실수냐?"**
→ Yes 일 때가 많고 — 그럴 때 즉시 사과 + 룰 문서화. 같은 함정 두 번 빠지면 룰이 부족했다는 뜻.

### 핵심 약속

이 매거진은 **2026-05-10 출발 후 수정 불가**. 그래서 출발 전에:
- 모든 P0 검증 통과 (`VERIFICATION.md`)
- 가족 4명 모두 폰에서 한 번씩 열어봄
- export JSON 백업 1회

출발 후엔 매거진은 "기록 매체" 로 전환. 가족이 흔적 남기고 (별점, 메모, 사진) export JSON 으로 가지고 옴. 그때부터는 메인테이너가 매거진 콘텐츠 손댈 일 없음.

### 가장 큰 학습 — "적은 게 많다"

스택 정해질 때 형이 했던 결정: vanilla HTML + CSS + JS 단일 파일.

- 5년 후에도 그대로 열림 (호환성)

이 결정이 매거진 메이커 + 메인테이너 둘 다 살림.

## § 9. Hidden Mission — 11살 도파민 설계와 state 영속성 함정

> 가족 본딩 + 교육 + 게임화 = 11살 조카가 좋아하면서도 친구한테 자랑할 수 있는 보너스 미션 6개. 작은 기능이었는데 회귀가 4번 났다.

### 디자인 결정 (왜 6개?, 왜 그린톤?, 왜 별도 페이지?)

**5 vs 7 vs 10 미션 개수 — 6 으로 합의**:
- 5 = 4일 여행에 1.25개/일 = 너무 빠름, 마지막 날엔 할 게 없음
- 10 = 너무 부담, 11살한테 "왜 이렇게 많아?" 거부감
- 7 → 6 = "현지 상점 직접 주문" BOSS 미션 추가, 가족 + 문화 + 학습 + 도전 골고루
- 1.5개/일 = 매일 하나 발견하는 재미, 마지막 날에도 1~2개 여유

**메인 9 vs 히든 6 톤 분리**:
- 메인 = 골드 + "TRIP CONQUERED · 오키나와 정복!" + 도-미-솔-도
- 히든 = 그린 + "FAMILY LEGEND · 패밀리 레전드!" + 미-솔-시-미
- "Master" 가 학원 톤이라 "Legend" 로 변경 — 11살이 친구한테 자랑할 수 있는 단어
- 골드와 그린 톤 차이로 "이건 별개 시스템" 자연스러운 인지

**별도 페이지 vs 통합 vs 분산**:
- 통합 (마무리 페이지에 같이) = 메인 9개 시각 강조 흐림
- 분산 (각 Day 안에 끼워넣기) = 11살이 미션을 찾으러 다녀야 함
- 별도 'hidden' 탭 = "한 곳에 모여있어 하나씩 클리어" 명확. 11살이 가장 이해 쉬움

### 트리거 — D1 호텔 도착 ✓ 시 unlock

**왜 d1-hotel 인가**:
- 너무 일찍 (prep 미션) = 매거진 진입하자마자 보너스, 메인 시스템 익숙해질 시간 없음
- 너무 늦게 (D3) = 4일 중 절반 지나서야 추가 미션 발견 = 늦음
- d1-hotel = 호텔 첫 도착 = "여행 시작 직후 풀리는 비밀" 내러티브 자연스러움

### 회귀 4건 (한 미션 안에서)

#### 회귀 1 — 변수명 typo (`id` vs `missionId`)
- 코드 작성 시 wireMissionCards 안 변수가 `missionId` 인데 `id === 'd1-hotel'` 라고 씀
- `id` = undefined → 조건 항상 false → unlock 모달 영영 안 뜸
- **교훈**: 다른 함수 변수명에 의존할 때 자기 함수 안 변수명 다시 확인. linter 가 잡아줘야 할 영역인데 vanilla JS 라 X
- **fix** (commit 8d3fe20): `missionId === 'd1-hotel'` + 코멘트에 "변수명 주의" 명시

#### 회귀 2 — state.hiddenUnlocked 영속성 함정
- 한 번 YES 누르면 `state.hiddenUnlocked = true` 영구 저장
- 다음 d1-hotel 도장 시 `!state.hiddenUnlocked` 가 false → cinematic skip
- 개발 테스트할 때 "왜 자꾸 안 뜨지?" 하면서 state.hiddenUnlocked 의 영속성을 까먹음
- **교훈**: localStorage 영속 플래그는 "한 번 true → 영원히 true" 임을 잊지 말 것. 테스트 시 명시적 리셋 절차 필요
- **fix** (commit c2f6d47): d1-hotel 도장 취소 시 `state.hiddenUnlocked = false` 자동 리셋 + confirm 메시지에 안내

#### 회귀 3 — Hidden 탭 가시성 3단계 미적용
- CSS 만 바꾸고 JS `updateHiddenUnlockUI()` 안 갱신해서 탭이 항상 안 보임
- 단계 1/2/3 enum 으로 정리:
  ```
  단계 1 (d1-hotel done X)         → display: none
  단계 2 (d1-hotel done O, !unlock) → 자물쇠 + 반짝
  단계 3 (hiddenUnlocked O)        → 라벨 + NEW
  ```
- **교훈**: 가시성 로직은 CSS + JS 양쪽 동기 필요 — 한 쪽 만지면 다른 쪽 함께 갱신
- **MISSIONS.md** 에 state machine 표 명시

#### 회귀 4 — Family Master vs 11살 톤
- 처음 "Family Master" 라 라벨링 → 학원 톤, 11살이 자랑하기 어색
- 사용자 피드백: "마스터는 좀 유치하다. 친구한테 자랑할 만한 톤"
- **교훈**: 단어 1개 차이가 전체 느낌 결정. 메인 사용자(11살) 어휘로 검수
- **fix**: "FAMILY MASTER" → "FAMILY LEGEND" / "패밀리 레전드"
- 동시에 "🎫 BONUS · For 조카 Only ✺" → "🎮 SECRET LEVEL · 너만 풀었어" (어른 톤 → 게이머 톤)

### 사운드 디자인 — "삐이익" 함정

처음 fireTicketIssued 사운드:
- sawtooth sweep 200→1200Hz (0.4s) — "삐이익!"
- + sine chime 도-솔-도
- + 저음 boom 60→40Hz

사용자 피드백: "소리가 너무 깜짝 놀라게 한다. 띠링 정도면 괜찮겠다."

→ sweep + boom 제거, sine chime 도-미-솔만 (0.3s, 볼륨 ↓)

**교훈**: 11살 알림 사운드 = "기쁨 알림 (띠링)" ≠ "긴급 알림 (삐이익)". 동일 컴포넌트도 톤이 어긋나면 사용자가 회피.

### 도파민 3단계 흐름 (최종 확정)

```
[click ✓ 호텔 도착!]
   ↓ 0~1.8s: MISSION COMPLETE (앰버, 일반 미션)
   ↓ 2.1s:   🎫 TICKET ISSUED cinematic (그린 confetti + 띠링)
   ↓ 3.8s+:  YES/NO 모달 ("남건우님께서 Hidden Mission을 활성화시키셨습니다")
```

각 단계가 별개 임팩트 — 한 번에 너무 많은 정보 주지 않고 자연스럽게 단계 상승.

## § 10. Place & Action 리스트 + 별 아이콘 정렬 — "별이 동그라미보다 옆으로 비뚤어 보이는 이유"

> 마무리 페이지에 "내가 다녀온 곳" 리스트를 만들었는데, 미션은 ★ / 일반 옵션은 ● 로 시각 구분. 그런데 별이 옆으로 살짝 어긋나 보이는 회귀가 났다.

### 사용자 요청 — "동그라미가 약간 우측으로 틀어져있네"

11살 조카·운전자·어른 모두 보는 마무리 리스트:
```
★ 도착! Naha             ← 미션 (별)
● 12:30 Blue Seal        ← 옵션 (동그라미)
★ Check-in 15:00         ← 미션 (별)
● 18:00 Sunset Beach     ← 옵션 (동그라미)
```

조카가 "어 이상하다, 별이 동그라미보다 약간 왼쪽이네?" 라고 하면 — 매거진의 "디테일이 좋다" 라는 인상이 깨짐. 폰트 디자인 이슈인데 사용자는 그 사실을 모르고 그냥 "허술해 보인다" 로 받아들임.

### 비유 — 핀과 스티커의 차이

**동그라미** 는 도화지에 핀을 꽂은 것. 핀이 **꽂힌 위치 = 정확한 지점**. 8mm 핀이라면 그 핀의 중앙이 정확히 "내가 정한 그 점".

**별 (★)** 은 스티커를 붙인 것. 스티커는 직사각형 종이에 별 그림이 있는데, **스티커 가장자리 = 별 가장자리** 가 아님. 스티커 좌측에 흰 여백이 있고 (left bearing 이라고 부름), 그 안에 별이 그려져 있음. 즉:

```
스티커 박스: |    ★    |
   여백        ↑
    └─ 이 여백 폭이 폰트마다 다름!
```

내가 "스티커를 좌상단 (-19, 14) 에 붙여" 라고 명령해도, 실제로 별의 시각 중심은 (좌측 여백만큼 + 별 폭/2) 만큼 오른쪽에 가있음. **폰트가 바뀌면 여백이 바뀌고 → 별 위치가 변함**. 사용자 OS 가 별 글리프를 어느 폰트로 fallback 하느냐에 따라 매번 다름.

### 1차 시도 — 좌표 미세조정 (실패)

내가 처음 한 일: 별 `left: -22px` 에서 `-23px` 으로 1px 옮김. 폰트 메트릭 보정해보겠다는 시도.

**왜 실패**: 그 폰트에선 1px 보정이 맞을지 몰라도 — Apple 폰트, Android 폰트, 한국 OS Galmuri 폰트, 일본 OS Yu Gothic 폰트 모두 별 글리프 left bearing 이 다름. 한 폰트에 맞추면 다른 폰트에선 또 어긋남. **폰트 의존 = 매번 다른 결과**.

### 2차 시도 — Flex Centering 로 폰트 의존성 제거 (성공)

```css
.visited-list .vl-item.mission::before {
  content: '★';
  width: 8px;          /* dot 와 똑같은 크기 박스 */
  height: 8px;
  top: 14px;           /* dot 와 똑같은 위치 */
  left: -19px;         /* dot 와 똑같은 위치 */
  display: flex;       /* 박스 안에서 강제 중앙 정렬 */
  align-items: center;
  justify-content: center;
  overflow: visible;   /* 박스 밖으로 별이 삐져나와도 OK */
  font-size: 18px;     /* 별 글리프 크기 — 박스보다 큼 */
}
```

**핵심 아이디어**:
- 동그라미와 **완전히 동일한 8×8 박스** 좌표 사용 (left:-19, top:14)
- 그 박스 안에서 `display: flex` + center/center → 별 글리프가 **박스의 기하학적 중심**에 강제로 정렬
- `overflow: visible` 로 18px 별이 8px 박스 밖으로 나와도 잘리지 않음

**비유**: 8×8 핀 자리에 박스를 그려놓고, 그 박스의 정확한 가운데에 스티커를 붙이는 작업을 자동화. 스티커 종이 폭이 어떻든, 별 그림 자체의 중심을 박스 중심에 맞춤. **폰트가 바뀌어도 박스는 안 바뀌니까 항상 정렬됨**.

### 일반 룰 — "텍스트 글리프를 픽셀 정밀도로 배치할 때"

| 상황 | 권장 방법 |
|---|---|
| 한 줄 텍스트 자연 흐름 (문장 안 단어) | 그냥 inline 흐름 — 메트릭 의존 OK |
| 다른 시각 요소와 픽셀 단위 정렬 (★ ↔ ●, 아이콘 ↔ 라벨) | **고정 박스 + flex centering** |
| 버튼/배지 안에 단일 글리프 (✓, ×) | 마찬가지로 flex centering |

**한 줄 룰**: 글리프 위치를 픽셀 정밀도로 다른 시각 요소와 맞춰야 한다면 — 폰트 메트릭에 의존하지 말고 박스로 가둬라.

### Canvas 동시 갱신 (UI ↔ Canvas Parity)

다운로드 PNG 에서도 미션 ★ / 옵션 ● 구분 필요. Canvas 는 `textBaseline: 'middle'` + `textAlign: 'center'` + `arc()` 동일 좌표 사용 — 자동 정렬되어 별도 보정 불필요.

```js
if (it.isMission) {
  ctx.fillText('★', 110, y + itemH/2);  // 텍스트 — 자동 center
} else {
  ctx.arc(110, y + itemH/2, 8, 0, 2*Math.PI);  // 도형 — 정확 center
  ctx.fill();
}
```

같은 좌표 (110, y+itemH/2) 에 두 방식 다 그리지만 — Canvas API 가 자동 처리.

### 1.3x 별 크기 확대 — "친구한테 자랑할 때 잘 보이게"

처음 별 14px / 동그라미 8px → 별이 동그라미보다 살짝 큰 정도. 사용자가 "별을 1.3배 키워서 더 시인성 좋게" 요청.

- font-size: 14 → 18 (1.286× = 사실상 1.3×)
- text-shadow 추가 (`0 1px 2px rgba(0,31,102,0.18)`) — 흰 배경에서 별 윤곽 살짝 떠 보임
- Canvas: 22px → 29px

**왜 효과 있었나**: 일반 옵션 (8px 동그라미) 보다 **별 글리프 가시 폭 ~14px → 거의 2배 큼**. "이건 미션이야!" 라는 시각 우선순위 명확.

### 한 줄 요약

> **폰트 글리프와 도형을 픽셀 정렬할 땐 박스로 가둬라. 그 박스 안에서 flex 가 알아서 가운데에 놓는다.**

## § 11. PRO TRAVELER 인증 카드 마무리 — "다운로드 받았는데 뭔가 빠진 느낌"

> 6 개 시크릿 미션 다 클리어 → 그린톤 인증 카드 (PRO TRAVELER) 다운로드 가능. 사용자 피드백 4 건이 한꺼번에 와서 한 세션에 다 처리한 이야기.

### 회귀 1 — 다운로드 PNG 에 오키나와 SVG 누락

**증상**: 사용자가 PNG 다운로드 후 "트로피 위에 오키나와 사진이 적용이 안되어있다."

**비유**: 학교 졸업장에 학교 로고가 있어야 하는데 — 화면 미리보기엔 있고, 인쇄해서 받아보면 로고만 빠진 채 출력. 같은 기관에서 만들어준 줄 알았던 두 결과물이 "어딘가 다르게 보임".

**원인**: 화면 UI 와 다운로드 PNG 가 **두 개의 다른 코드** 로 그려짐:
- 화면: HTML/CSS — `<svg><image href="okinawa-mainland.svg"></image></svg>` 그대로 브라우저가 렌더
- PNG: JavaScript Canvas — `ctx.fillText('🏆')` 로 트로피만 그리고 SVG 그리는 단계 자체 누락

→ § 7 "UI ↔ Canvas Parity" 룰 위반의 또다른 사례. 문서엔 룰이 있는데 실제 작업 시 까먹음.

**해결**:
1. `drawOkinawaSilhouetteToCanvas(ctx, cx, topY, size)` 헬퍼 신설 — SVG 비동기 로드 → off-screen canvas 에 cream 톤 입혀서 그리기
2. `downloadLegendImage()` 를 `async` 함수로 변환 + `await drawOkinawaSilhouetteToCanvas(...)` 추가
3. 트로피·텍스트·통계 모든 y 좌표 재배치 (오키나와 위에 올라감)

**비유**: 졸업장 공장에 "로고 도장 찍는 단계" 추가. 도장이 마르길 기다린 뒤 (`await`) 다음 단계 진행.

### 회귀 2 — "메인 9개 + 시크릿 6개" 라는 잘못된 멘트

**증상**: 6 개 시크릿 미션 완료 시 풀스크린 알림에 "메인 9개 + 시크릿 레벨 6개 = 진짜 잘 했어" 텍스트.

**사용자 피드백**: "Hidden mission 은 컨셉상 메인과 독립적이게끔 분리되어 있는데, 보상 멘트에선 둘이 합쳐져 있다."

**왜 문제**:
- Hidden Mission 의 핵심 가치 = **메인과 독립** (CONCEPT.md 페르소나에서 11살이 친구한테 자랑할 별도 트랙)
- 멘트가 "9 + 6" 으로 합치면 → 사용자가 "메인 안 하면 시크릿 못 받음" 으로 오해
- 컨셉이 멘트 한 줄에 흐려짐

**해결**: "시크릿 레벨 6개 = 진짜 잘 했어" — 메인 9 언급 제거.

**교훈**: 핵심 컨셉이 사용자에게 전달되는 곳이 100 군데면 — 99 군데 맞아도 1 곳 어긋나면 컨셉 흐려짐. 멘트 점검 시 모든 등장 위치를 grep 으로 한 번에 훑고, 컨셉과 어긋나는 표현 찾아내기.

### 회귀 3 — 형광펜 마커 강조 ("자랑스럽게, 학원 톤 X")

**증상**: 인증 메시지 안 핵심 문구 `<strong>여행에서 제 몫을 한 사람</strong>` 이 그냥 검정색 굵은 글씨로 평범.

**사용자 요청**: "글자 엔터랑 중요 내용을 자연스러운 톤으로 강조."

**디자인 옵션 분석**:
- 박스 처리 (네모 칸 + 배경) → 광고 배너 같음, 차분한 가족 톤 깨짐
- 컬러만 변경 (예: 빨강) → 경고 느낌, "잘 했어" 톤 안 맞음
- 굵게 + 이탤릭 → 형식적, 흐름 끊김
- **형광펜 marker 효과** → "학교 노트 정리, 시험 전날 중요한 거 그어놓음" 느낌. 11살 친화

**선택**: 형광펜 (highlighter)
```css
.fm-body strong {
  color: #0e6b56;  /* deep green — 헤드라인 톤 */
  background: linear-gradient(180deg,
    transparent 62%,
    rgba(255, 196, 0, 0.55) 62% 92%,   /* 노란 띠: 글자 하단 30% */
    transparent 92%);
  padding: 0 3px;
  border-radius: 2px;
}
```

**비유**: 학교에서 시험 공부할 때 형광펜으로 중요한 부분만 그어놓는 그 느낌. 글자 색은 검정 그대로 유지하고 (가독성 ↑), 노란 띠가 "이 부분이 핵심" 표시. 전체를 색칠하지 않고 **글자 아래 30% 영역에만** 띠가 들어가서 — 자연스럽게 "주의 환기" 됨.

**왜 effective**: 색을 통째로 바꾸면 글씨가 안 읽힐 수 있음. 형광펜 띠는 **배경에 깔리지 글자 가독성은 그대로**. "강조 안 한 척 강조" 라는 미세한 톤.

### 회귀 4 — 줄바꿈이 어색한 위치에서 끊김

**증상**: 자동 wrap 으로 "여행에서 제 몫을 한" / "사람" 이 두 줄로 갈라져서 형광펜 띠가 중간에서 잘림.

**사용자 요청 1차**: 자연스러운 줄바꿈 + 호흡 추가.

**사용자 요청 2차** (구체화): 4 개 액션을 각자 한 줄씩 분리:
```
일본에서 혼자 점원한테 주문하고,
가족 짐 들어주고,
류큐 시사 찾아내고,
추라우미 물고기 그림으로 설명한 너는       ← '너 —' 가 아닌 '너는'
[여행에서 제 몫을 한 사람]이라는 뜻이야.    ← 형광펜 강조
친구한테 자랑해도 돼.
```

**핵심 변화**: `너 —` (영문 em dash) → `너는` (한국어 자연 어미). em dash 는 영문에선 자연스러운데 한국어 흐름엔 어색. "너는 [형광펜] 사람이라는 뜻이야" 가 매끄럽게 이어짐.

**기술 안전망**: `word-break: keep-all` 추가 — 좁은 화면 (모바일) 에서도 한국어 어절 (`제 몫을`, `한 사람`) 이 절대 글자 사이에서 끊기지 않음. 형광펜 띠가 글자 중간에서 잘리는 사고 차단.

### 페이지네이션 간격 — "박스 밑에 너무 많이 떨어져 있다"

**증상**: 인증 메시지 박스 밑과 "← 이전 / 다음 →" 페이지 버튼 사이 80px 의 빈 공간.

**사용자 요청**: "지금이 10이라고 한다면 한 3 정도로."

**원인**:
- `.page-pagination` `margin-top: 40px` (박스와 디바이더 라인 간 간격)
- `.page-pagination` `padding-top: 40px` (디바이더 라인과 버튼 간 간격)
- 합 80px. 사용자가 보기엔 "그냥 휑한 빈 공간".

**해결**:
- margin-top 40 → 8 (-80%)
- padding-top 40 → 12 (-70%)
- 합 20px. 약 25% 수준.

**왜 그렇게 띄어져 있었나**: 매거진 톤이라 "여백 = 고급" 같은 인식. 하지만 모바일에선 **여백이 정보 없는 공간** → 사용자가 "끝났나? 더 있나?" 헷갈림. 페이지 끝 안전 여백 (60px padding-bottom) 은 그대로 유지하면서 위쪽 거품만 컷.

**룰**: 매거진 디자인의 "고급스러운 여백" 은 종이 잡지 기준. 모바일 화면에선 1/3 ~ 1/2 수준으로 줄여도 충분히 호흡 있음.

### 한 줄 요약

> **인증 카드 같은 "공유 자산" 은 화면 + 다운로드 + 텍스트 + 강조 + 줄바꿈 모두 하나의 일관된 톤으로 점검해야 한다. 4 곳 중 한 곳만 어긋나도 사용자는 "허술해 보임" 을 감지한다.**

## § 12. 11살 사용자 톤 디자인 — 작은 결정 30개의 누적

> 메인 사용자가 11살 (남자 조카) 인 매거진. "친구한테 자랑할 만한" 톤이 페르소나의 핵심. 이번 세션에서 누적된 작은 결정들을 모음.

### 어휘 — "마스터 → 레전드 → 프로 트래블러"

| 단어 | 톤 | 11살 친구 자랑 | 결정 |
|---|---|---|---|
| Family **Master** | 학원, 시험 1등 | "엄마 자랑할 만" — 친구 X | ❌ 너무 유치 |
| Family **Legend** | 게임 (디아블로, 오버워치) | "친구한테 잘 통함" | 🟡 1차 채택 |
| **PRO TRAVELER** | 일반 명사 + 영문 | "어른스러우면서 부담 X" | ✅ 최종 |

**선택 이유**: "Legend" 도 좋지만 게임 안 하는 친구한테는 의미 약함. "PRO TRAVELER" 는 누구한테 보여줘도 의미 통함 + "여행 = 어른의 영역" 이미지가 11살한테 살짝 우쭐함.

**한 줄 룰**: 메인 사용자 (11살) 의 친구 단톡방에서 자연스럽게 통하는 단어 → ✅. 학원 칭찬 톤 → ❌.

### 사운드 — "삐이익" vs "띠링"

처음 fireTicketIssued 사운드 디자인:
- sawtooth sweep 200→1200Hz (0.4s) — 강한 "삐이익!"
- + sine chime 도-솔-도
- + 저음 boom 60→40Hz

**비유**: 알람 시계 vs 카톡 도착음. 알람 시계는 "지금 일어나!" 강한 자극, 카톡 도착음은 "어 누구 왔네" 부드러운 환기. 둘 다 "알림" 이지만 톤 정반대.

**사용자 피드백**: "Yes 누르면 놀라겠다."

→ sweep + boom 제거, sine chime 도-미-솔 만 (0.3s, 볼륨 ↓). "띠링" 으로 톤 다운.

**룰**: 11살 알림 사운드 = "기쁨 환기 (띠링)" — "긴급 알림 (삐이익)" 아님. 도파민 = 강도 X, 빈도 + 의외성 O.

### 색상 — 메인 골드 vs 히든 그린 (자연 분리)

**왜 두 색**:
- 메인 미션 = 골드 (#ffc400) — 황금 트로피 톤, 공식 보상 느낌
- 히든 미션 = 그린 (#0e6b56 ~ #16a085) — 비밀, 자연, 성장 톤

**시각적 효과**: 사용자가 색만 봐도 "이건 다른 트랙이구나" 인지. 텍스트 안 읽어도 직관.

**비유**: 학교 행사장에서 졸업생 (검정 가운) vs 재학생 (파랑 가운) 색만으로 구분. 글자 안 봐도 알아봄.

### 시각 아이콘 — 별 vs 동그라미

방문한 곳 리스트 좌측 bullet:
- 미션 = ★ (성취, 행동)
- 옵션 = ● (장소, 들른 곳)

**비유**: 카드 게임에서 일반 카드 vs 스페셜 카드. 같은 게임이지만 즉시 구분 가능.

**왜 단순한 모양 차이로 충분한가**: 11살은 시각 우선. 작은 아이콘 차이가 "이건 다르네" 인식 → 자세히 안 봐도 OK.

### 어미 — "너 —" vs "너는"

영문 em dash (`—`) 는 영어 글에서 자연스럽지만 한국어에선:
- "너 —" 어색, 스타카토
- "너는" 매끄러움, 한국어 자연 어미

11살은 영어 글 자주 안 읽음 → "너 —" 하면 멈춤. "너는" 하면 술술.

**룰**: 영문식 punctuation (em dash, semicolon, ellipsis) 는 한국어 본문에선 90% 자연 어미로 변환.

### 강조 — "굵게" vs "형광펜"

`<strong>` 의 기본 렌더 = 검정 굵게. 너무 흔해서 강조 효과 약함.

**대안 디자인 비교**:
| 방식 | 11살 인상 |
|---|---|
| 검정 굵게 | "그냥 굵은 글씨" |
| 빨간색 | "혼나는 느낌" — 부정적 |
| 노란 박스 | "광고" — 신뢰도 ↓ |
| 형광펜 띠 (선택) | "학교에서 그어놓은 그거" — 친근 + 중요 |

**구현**: linear-gradient 로 글자 하단 30% 에만 노란 띠. 글자색은 그대로 (deep green). 가독성 + 강조 동시.

### 줄바꿈 — "낭독 호흡" 으로 디자인

긴 문장을 한 줄로 두면 자동 wrap → 어색한 위치에서 끊김. 11살이 읽다가 "어디까지가 한 의미야?" 헷갈림.

**해결 패턴**: 의미 단위로 명시적 `<br>`:
```
일본에서 혼자 점원한테 주문하고,
가족 짐 들어주고,
류큐 시사 찾아내고,
추라우미 물고기 그림으로 설명한 너는
```

각 줄 = 한 가지 액션 = 한 호흡. 4 개 액션이 한 줄씩 떨어지면서 "와, 이걸 다 했네" 카운트업 느낌.

**비유**: 시 낭독 vs 산문 낭독. 같은 내용이라도 줄바꿈이 호흡을 결정. 시는 한 줄 = 한 단위. 자랑 메시지도 마찬가지.

### 컨텐트 vs 단어 — "곳 vs 건"

방문 리스트 카운트 단위:
- "15곳" → 장소 만 셀 때 (Blue Seal, Sunset Beach 등)
